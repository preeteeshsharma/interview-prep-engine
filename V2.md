# V2: Multi-User Support

V1 is a single-owner deployment — one set of credentials, one WhatsApp number, one prep-vault.
V2 makes one deployment serve many users. The codebase is architected to make this a
minimal change.

---

## What V2 adds

### 1. `User` table (new migration)

```python
class User(Base):
    __tablename__ = "users"
    email: Mapped[str]            # PK — human-readable identity
    whatsapp_phone: Mapped[str]   # linked after registration
    github_token: Mapped[str]     # user's own token (repo scope)
    vault_repo: Mapped[str]       # e.g. "alice/prep-vault"
    registered_at: Mapped[datetime]
```

### 2. `user_email` FK on existing tables (new migration)

Add `user_email: Mapped[str]` (FK → `users.email`) to:
- `Interview`
- `PrepPlan`
- `MockSession`
- `WeakPattern`

All repo queries gain a `user_email` filter. No business logic changes.

### 3. Registration flow (email)

User sends an email to `prep@yourdomain.mailgun.org`:

```
Subject: register
Body:
  github_token: ghp_abc123
  vault_repo: myname/prep-vault
```

`/hooks/inbox` detects `subject == "register"`, creates a `User` row keyed by
sender email, replies with a linking code.

### 4. WhatsApp linking (one new intent)

```
User texts:  link their@email.com
Engine:      Linked. Text 'prep <company>' to get started.
```

Stores `whatsapp_phone` on the `User` row. From this point, all inbound
WhatsApp messages resolve to a `UserContext` via `get_user_context(sender_phone)`.

### 5. Morning drill cron

V1 iterates over all active interviews (single owner).
V2 groups interviews by `user_email`, fetches `UserContext` per user, sends to
their WhatsApp, commits to their vault.

## WhatsApp: Sandbox → Production

- Free, no Meta approval needed
- Users must send `join <code>` once to activate
- Only works for numbers that have joined — not suitable for real users
- Sandbox number is shared across all Twilio developers (`+1 415 523 8886`)

### V2: WhatsApp Business API
- Apply for a WhatsApp Business Account (WABA) via Twilio or Meta directly
- Get a dedicated phone number for your service
- Users message your number directly — no join code, no sandbox
- Meta approval takes 1–2 business days (straightforward for a utility app)
- **No code changes needed** — same webhook, same `sender` capture, same

**How to apply:**
1. Twilio Console → Messaging → WhatsApp → Senders → Request access
2. Or: Meta Business Manager → WhatsApp → Add phone number
3. Twilio route: simpler, they handle Meta verification on your behalf

## Infrastructure for V2

### Current (V1): Fly.io free + Supabase free
- 1 Fly.io machine, 256MB RAM, Singapore region
- Supabase Postgres free tier (500MB, 60 connections)
- Works for 1 user comfortably

### V2 (small multi-user, ~10–50 users):
- **Fly.io**: upgrade to `shared-cpu-1x` with 512MB RAM or add a second machine
  (`fly scale count 2`). Still cheap (~$5–10/month).
- **Supabase**: free tier handles this load. Upgrade to Pro ($25/month) only if
  you hit the 500MB storage limit or need the pause-disable feature.
- **APScheduler**: runs on one machine only — add a leader-election flag or
  migrate to Fly.io Cron if scaling to multiple machines.

### V2 (larger scale, 100+ users):
- Migrate morning drill from APScheduler to a proper task queue (Celery + Redis,
  or Fly.io Machines as one-off tasks)
- Add Supabase Pro for connection pooler (PgBouncer) and higher connection limits
- Consider read replicas for transcript queries

### SQLAlchemy connection pool at scale
Current: `create_async_engine(..., pool_pre_ping=True)` with default pool (5 connections).
At 50+ concurrent users: set `pool_size=10, max_overflow=20` and use Supabase's
session pooler (port 5432 on `pooler.supabase.com`) to stay within the 60-connection
free tier limit.

## Problem Corpus (DSA / LLD / Sysdesign)

### V1 (now): Claude's built-in knowledge
`generate_plan` asks Claude to pick problems directly from its training knowledge:
- **DSA** → LeetCode problems by name and number (e.g. "Two Sum #1", "Sliding Window Maximum #239")
- **LLD** → standard catalogue: parking lot, library system, chess, elevator, etc.
- **Sysdesign** → company-tailored: Zapier → webhook delivery, Stripe → payment processing

No corpus file, no local paths, no machine-specific setup. Works out of the box.

### V2 option: private corpus via `match_corpus`
`app/tools/match_corpus.py` is a documented stub — not wired by default.
For users who maintain their own curated problem sets (e.g. a private LLD repo):

1. Set `CORPUS_ROOTS` in `.env`:
   ```
   CORPUS_ROOTS=/path/to/lld-repo:LLD,/path/to/dsa-repo:DSA
   ```
2. Implement `match_corpus()` to scan those paths and return matching problem names
3. In `generate_plan()`, append `match_corpus()` results to the Claude prompt

This is optional — Claude's built-in knowledge covers LeetCode comprehensively
and does not require any local files.

### LeetCode coverage
Claude knows the full LeetCode problem set through its training cutoff. It picks
problems appropriate to the company and round type:
- Top-N problems for the company's domain (graph problems for a maps company, etc.)
- Avoids problems recently sent (tracked via `exclude_recent` in `generate_plan`)
- Weighted toward weak patterns identified from past sessions

## Security considerations for V2

- **GitHub tokens**: store encrypted at rest (Supabase column encryption or
  application-level AES before insert). Never log them.
- **Rate limiting**: add per-sender rate limiting on the Twilio webhook to prevent
  abuse (e.g. 10 messages/minute per number).
- **Registration validation**: verify the sender email domain if restricting to
  specific organisations (e.g. company email only).
- **Vault isolation**: each user's `vault_repo` is their own private repo — data
  is naturally isolated at the GitHub level. No cross-user data access possible
  through the vault.
- **Anthropic key storage**: if storing per-user keys, encrypt before insert.
  Consider letting users bring their own key via a secure link (e.g. one-time
  setup URL) rather than email.

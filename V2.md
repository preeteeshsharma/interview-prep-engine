# V2: Multi-User Support

V1 is a single-owner deployment — one set of credentials, one WhatsApp number, one prep-vault.
V2 makes one deployment serve many users. The codebase is architected to make this a
minimal change.

---

## Touch points (all stubbed in V1 — replace bodies only in V2)

### 1. `app/lib/user_context.py` — identity resolution

```python
async def get_user_context(sender_phone=None, email=None) -> UserContext:
```

V1: ignores both params, always returns owner from settings.  
V2: looks up `User` table by phone (Twilio) or email (Mailgun). Replace body only.  
All callers already pass the right identifier — no caller changes needed.

---

### 2. `app/routes/webhooks/inbox.py` — registration + sender lookup

Two stubs already in place:

```python
def _is_registration(payload) -> bool: ...       # detects "register" subject
async def _handle_registration(payload) -> None: # creates User row, replies
```

V2: implement `_handle_registration` to parse `github_token` + `vault_repo`
from the email body and call `UserRepository.create()`.

`get_user_context(email=payload.sender)` is already called — V2 activates
automatically once the User table exists.

---

### 3. `app/routes/webhooks/twilio.py` — per-user context + link intent

```python
async def _handle_link(sender, args) -> str: ... # stub — logs not_implemented
```

`get_user_context(sender_phone=sender)` is already called in `_route()`.

V2: implement `_handle_link` to call `UserRepository.link_phone(email, sender)`.

---

### 4. `app/jobs/morning_drill.py` — user loop

```python
async def _get_active_users() -> list[tuple[str, UserContext]]:
```

V1: returns single owner. V2: replace body to query `User` table.  
The interview loop already iterates over the result — no loop changes needed.

## What V2 adds

### 1. `User` table (new migration)

```python
class User(Base):
    __tablename__ = "users"
    email: Mapped[str]            # PK — human-readable identity
    whatsapp_phone: Mapped[str]   # linked after registration
    github_token: Mapped[str]     # user's own token (repo scope)
    vault_repo: Mapped[str]       # e.g. "alice/prep-vault"
    registered_at: Mapped[datetime]
```

### 2. `user_email` FK on existing tables (new migration)

Add `user_email: Mapped[str]` (FK → `users.email`) to:
- `Interview`
- `PrepPlan`
- `MockSession`
- `WeakPattern`

All repo queries gain a `user_email` filter. No business logic changes.

### 3. Registration flow (email)

User sends an email to `prep@yourdomain.mailgun.org`:

```
Subject: register
Body:
  github_token: ghp_abc123
  vault_repo: myname/prep-vault
```

`/hooks/inbox` detects `subject == "register"`, creates a `User` row keyed by
sender email, replies with a linking code.

### 4. WhatsApp linking (one new intent)

```
User texts:  link their@email.com
Engine:      Linked. Text 'prep <company>' to get started.
```

Stores `whatsapp_phone` on the `User` row. From this point, all inbound
WhatsApp messages resolve to a `UserContext` via `get_user_context(sender_phone)`.

### 5. Morning drill cron

V1 iterates over all active interviews (single owner).  
V2 groups interviews by `user_email`, fetches `UserContext` per user, sends to
their WhatsApp, commits to their vault.

## What does NOT change in V2

- `Interviewer`, `Observer`, `Coach`, `MockOrchestrator` — untouched
- `classify_rounds`, `generate_plan`, `record_completion` — untouched
- Twilio/Mailgun webhook infrastructure — one new intent (`link`), one new
  registration path in inbox
- `commit_file()` — already accepts `github_token` and `vault_repo` as params
- Anthropic API — host pays, single key in env (or add `anthropic_api_key` to
  `User` table if you want per-user billing)

---

## Cost model for V2

| Who pays | What |
|---|---|
| Host | Anthropic API, Twilio number, Railway |
| Each user | Their own GitHub token + private vault repo |

If you want per-user Anthropic billing: add `anthropic_api_key: Mapped[str | None]`
to `User`; `get_user_context()` returns it; `anthropic_client.py` already accepts
a token parameter (add one if not there yet).

---

## Migration path from V1

1. `alembic revision --autogenerate -m "add users table"`
2. `alembic revision --autogenerate -m "add user_email fk to core tables"`
3. Seed one `User` row for the existing owner (data migration)
4. Replace `get_user_context()` body (5 lines)
5. Add `register` detection to inbox webhook (~20 lines)
6. Add `link <email>` intent to Twilio router (~10 lines)

Estimated effort: ~half a day. Zero changes to agents, tools, or DB repos beyond
the new FK filter.
